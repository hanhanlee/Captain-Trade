## 第十七章：資料管理 — 快取與背景抓取

### 為什麼需要本機快取？

FinMind 免費帳號每小時只有 600 次 API 請求；Sponsor 帳號可用額度較高，目前系統會依 `user_info` 回傳額度調整，Sponsor 盤後/非交易時間可用到每小時 6000 次。全市場約 950 檔普通股，每次掃描若都從網路抓取，很快就會超出限制。

系統的解法是：**把歷史股價存在你的電腦裡**（本機快取），掃描時先讀本機資料，只有遇到新資料才呼叫 API。

### 背景預抓取工作器

App 啟動後，系統會自動在背景開始下載全市場的歷史價格資料。工作器由三條並行執行緒組成：

| 執行緒 | 任務 | 週期 |
|--------|------|------|
| 主執行緒 | OHLCV 核心價格（含 FinMind 全市場批次抓取 + 逐檔補完）、Yahoo Bridge | 持續執行 |
| 法人執行緒 | 三大法人買賣超（`TaiwanStockInstitutionalInvestorsBuySell`，all_by_date） | 每 5 分鐘輪詢 |
| 融資執行緒 | 融資融券（`TaiwanStockMarginPurchaseShortSale`，all_by_date） | 每 5 分鐘輪詢 |

你不需要做任何事，三條執行緒會靜靜地跑在背景。

| 時段 | 每小時上限 | 說明 |
|------|----------|------|
| 交易時間（09:00–盤後切換前） | Free 100 次；Sponsor 3000 次 | Free 保守保留額度給手動操作；Sponsor 可在盤中持續補快取，但仍保留一半左右額度給手動查詢與即時快照 |
| 盤後 / 非交易時間 | Free 600 次；Sponsor 依帳號額度，最高 6000 次 | 全力填充快取 |
| 全速重建模式 | 依帳號額度 | 開放全部額度，手動啟動 |

系統不再固定用單一的盤後切換時間。它會記錄「當天盤後第一筆成功更新時間」，之後自動把全速開始時間往前提早 10 分鐘，逐步貼近資料源實際更新節奏。

### 執行緒即時狀態面板

「資料管理」頁面的工作器狀態區新增「執行緒即時狀態」三欄面板，讓你一眼看到三條執行緒各自的即時情況：

| 欄位 | 說明 |
|------|------|
| **主執行緒** | 目前正在抓取的股票代號、已完成/總數、OHLCV 批次是否完成 |
| **法人執行緒** | 已補完筆數、Stable Rounds（連續相同輪次）、下次確認倒數、有新資料時間、最後確認時間 |
| **融資執行緒** | 同法人執行緒格式 |

**兩種時間戳的差異：**

- **有新資料（last_new_data_at）**：最後一次 FinMind 有回傳新增資料的時間。這個時間可能長時間不變，代表 FinMind 目前尚未發布新筆數，不是程式異常。
- **最後確認（last_checked_at）**：最後一次執行緒實際送出 API 請求的時間。每 5 分鐘更新一次，讓你確認執行緒仍在正常運作。

**Stable Rounds（連續相同輪次）**：若連續數輪 API 都回傳相同的筆數，Stable Rounds 會遞增。這是觀察資訊，不會造成執行緒停止——執行緒會持續輪詢直到服務關閉。

### 休市模式

連假或停市期間，法人資料不會更新。開啟「🏖️ 休市模式」可讓法人快取永不過期，避免反覆呼叫 API 浪費配額。

- **開啟位置：** 「6 - 資料管理」頁面最上方的切換按鈕
- **效果：** 法人資料快取無限期有效，不消耗 API 額度
- **恢復交易後：** 手動關閉此開關，下次掃描時系統會自動更新法人資料

### 交易日判斷與參考日期

資料管理頁現在會先判斷「今天是否應該有交易資料」，再決定狀態文字與進度基準：

- 週六、週日會顯示非交易日，不會誤判成交易時間
- 開啟休市模式時，會視為手動休市，不會要求今日法人或融資資料
- 一般交易日會用 `resolve_latest_trading_day()` 推估最新應有資料日
- 下方「資料待更新」與進度條會以最新交易日為基準，不再只看工作器記憶體中的暫時計數

這樣可以避免週末看到「等待 FinMind 更新今日資料」或「交易時間中」這類誤導訊息。

### 快取品質診斷怎麼看

「🔍 快取品質診斷」會把本機資料分成 5 類，幫你快速找出為什麼某些股票掃不到、回測跑不動、或資料明顯不完整：

| 分類 | 意義 | 你應該怎麼做 |
|------|------|-------------|
| 完全缺失 | 股票清單裡有這檔，但本機完全沒有任何價格快取 | 用「手動補抓」或啟動重建模式 |
| 資料不足 | 有資料，但天數低於你設定門檻（例如少於 100 天） | 若要做掃描或回測，建議補齊 |
| 資料過舊 | 最新一筆資料離今天太久，可能停留在數天前 | 先刷新快取，再重新掃描 |
| 已下市 | 該股票在 `ref/suspendList.csv` 中，系統視為已下市或停止交易 | 不列入問題股票，不需要反覆補抓 |
| 全部快取摘要 | 顯示所有股票的最早/最新日期與天數分布 | 用來整體判斷資料庫健康度 |

這個診斷的目的不是把所有股票都補到一模一樣，而是讓你先把「真的會影響使用」的問題資料找出來。系統會把已下市股票排除在主要問題清單之外，避免你把配額浪費在不需要修復的代碼上。

### 市場指數資料（TAIEX）為什麼重要

回測模組現在會使用 **TAIEX（台灣加權股價指數）** 當作市場基準。這筆資料主要用在：

- 比較策略績效是否優於大盤
- 支援相對強弱（RS）或市場基準相關的計算
- 避免基準曲線空白，或因市場資料缺失導致報告不完整

如果資料管理頁看到「📈 市場指數下載（TAIEX）」尚未完成，建議先補齊，再執行正式回測。否則即使個股資料完整，部分報表或比較基準仍可能缺值。

**回測歷史重建的狀態顯示**

啟動「📼 回測歷史資料重建」後，頁面上方會直接顯示：

- 目前是否處於回測重建模式
- 回測待補股票數
- 正在抓取哪一檔（會顯示 `[回測] 股票代碼`）
- 回測歷史重建進度條
- 最近一次抓取時間

若遇到 429 限額，頁面也會顯示暫停倒數，可按「⚡ 立即恢復」提前繼續。

### 每日自動更新流程

每個交易日，背景工作器會自動依序完成以下階段：

| 時間 | 階段 | 說明 |
|------|------|------|
| 盤後（非交易時間，每日一次） | **① FinMind OHLCV 全市場批次** | 以 `all_by_date=True` 一次 API 呼叫取得全市場今日收盤價，完成後記錄已抓取日期，不重複執行 |
| 全天 | **② OHLCV 逐檔補完** | 持續補齊近 400 天 K 線資料（逐股），交易時間 Free 限速 100 次/小時、Sponsor 限速 3000 次/小時，盤後依帳號額度全速 |
| 13:45–FinMind 全速窗口前 | **③ Yahoo Bridge** | 只在收盤後早期、FinMind 尚未穩定更新前，以 Yahoo Finance 補充今日收盤價；進入 FinMind 全速盤後窗口後不再跑 Yahoo Bridge |
| 盤後持續輪詢（每 5 分鐘） | **④ 法人連續確認** | 法人執行緒每 5 分鐘以 `all_by_date=True` 抓取全市場法人資料；有新增筆數才寫入 DB 並記錄時間；無停止條件 |
| 盤後持續輪詢（每 5 分鐘） | **⑤ 融資連續確認** | 融資執行緒與法人執行緒並行，每 5 分鐘抓取全市場融資融券；邏輯相同 |
| 非交易時間 + OHLCV 完成後 | **⑥ Sponsor 主力券商背景補完** | 以「日期優先」往回補 `broker_main_force_cache`；**OHLCV 未完成時不會執行**，確保今日收盤資料優先 |
| 每日完成後 | **⑦ 基本面快取** | OHLCV 全部補齊後，自動抓取各股財報指標（90 天 TTL） |

> **交易時間額度調整：** Free 模式交易時間維持 100 次/小時，Sponsor 模式交易時間提高到 3000 次/小時；盤後與重建模式仍依帳號額度，Sponsor 預設可用到 6000 次/小時。此調整是為了讓 Sponsor 帳號在開盤期間仍能補快取，同時保留部分額度給手動查詢與即時快照。

> **Yahoo Bridge vs FinMind 備援的差別：**
> - **Yahoo Bridge**（③ 階段）：只在 13:45 到 FinMind 全速窗口前短暫補充今日收盤 OHLCV，讓資料管理頁可以較早看到今日資料，但**無法提供法人/融資**。若已進入 FinMind 盤後全速更新時間，系統會跳過 Yahoo Bridge，改由 FinMind 補資料。
> - **強制 Yahoo Finance**（手動開關）：當 FinMind 達到目前帳號每小時上限時的即時備援，此時法人條件停用。

資料管理頁的狀態卡會顯示目前所在的工作階段，讓你一眼看出系統在做什麼：

### 為什麼法人/融資資料比 OHLCV 晚？

法人籌碼與融資融券的資料源（FinMind）通常在盤後數小時才發布，且發布時間每日不固定。系統採用**持續輪詢**策略應對這種不確定性：

- 法人執行緒與融資執行緒從 App 啟動後即開始每 5 分鐘確認一次 FinMind 是否有新資料
- 只要 FinMind 回傳的筆數比上一輪多，就立即寫入資料庫並記錄時間
- **沒有「完成」條件**：系統無法確知 FinMind 當日最終會發布幾筆；執行緒會持續輪詢，收集發布時序資料供未來分析
- **額度評估**：Sponsor 帳號每小時 6000 次，法人 + 融資各 1 次/5 分鐘，全天僅消耗約 576 次，對總額度影響極小

因此，「法人 X 筆」的數字在盤後可能緩慢增加，這是 FinMind 分批發布的正常現象，不代表系統異常。

### 法人/融資連續確認機制

| 機制 | 說明 |
|------|------|
| 每 5 分鐘輪詢 | 法人與融資執行緒各自獨立每 5 分鐘呼叫 FinMind `all_by_date` API |
| 只記錄有新資料的時間 | 若此輪筆數 > 上一輪，才寫入 `finmind_fetch_timing` 資料表，避免 log 過多無意義記錄 |
| 最後確認時間（in-memory） | 每次 API 呼叫後不論結果都更新 `last_checked_at`，頁面顯示用，不寫入 DB |
| Stable Rounds（顯示用） | 連續相同筆數的輪次計數，僅供觀察，不會觸發停止 |
| 無停止條件 | 系統不宣告「今日法人/融資已完成」，持續確認直到服務關閉 |

這個設計的目的是：在 FinMind 發布時序不確定的前提下，確保只要 FinMind 發布資料，系統就能在最多 5 分鐘內更新到本機快取；同時收集每日首次拿到資料的時間，做為未來優化輪詢策略的依據。


### 主力券商補完排查紀錄（2026-04-20）

4. 2026-04-20 已修正欄位 mapping，並以 `scripts/repair_broker_main_force_cache.py` 清除壞快取與對應狀態。

若未來主力券商補完看似停住，優先檢查：

### 失敗股票與自動略過

若某檔股票長期抓不到新資料，系統會自動區分：

- **今日暫時失敗**：先標記 `suspended`，之後仍會再試
- **長期無新資料**：若舊快取已超過 180 天沒有更新，會視為可能下市或合併，改標記為 `delisted`
- **完全無資料**：例如特別股、權證或資料源未收錄，會從背景重試清單中排除

這個機制的目的是把 API 配額留給真正還有更新價值的股票。

### Yahoo Finance 補充機制

系統有兩種使用 Yahoo Finance 的情境，機制不同：

**① Yahoo Bridge（每日早期橋接，13:45 到 FinMind 全速窗口前）**

台股 13:30 收盤後，FinMind 可能尚未立即提供完整當日資料。Yahoo Finance 約 15 分鐘延遲即有收盤資料，工作器只會在 13:45 到 FinMind 全速更新窗口前，短暫批次補充全市場收盤 OHLCV，填入本機快取，讓盤後早期可以先用今日價格做掃描。進入 FinMind 全速盤後更新時間後，系統不再跑 Yahoo Bridge，避免 Yahoo 限流並改由 FinMind 正式資料補正。

- 補充僅限 OHLCV（開高低收量），**無法提供法人/融資**
- 使用 INSERT OR IGNORE，不影響後續 FinMind 正式資料覆蓋
- 若有股票在 Yahoo 查無資料（停牌、剛上市等），頁面會顯示可展開的失敗清單，可選擇「重新抓取」或「略過」

**② 強制 Yahoo Finance（手動開關 🔀）**

當 FinMind 達到目前帳號每小時上限，或你想強制切換時手動啟用：
- 頁面最上方的「🔀 強制 Yahoo Finance」切換即可
- **法人/融資條件自動停用**（Yahoo Finance 無此資料）
- 下一個小時額度重置後可手動關閉，恢復 FinMind
